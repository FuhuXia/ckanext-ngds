How to Submit Data to the NGDS

Information resources become part of NGDS when they are web-accessible and described by a metadata record in the NGDS metadata catalog. Metadata records in the NGDS catalog contain instructions used to access the data they describe.

Though hard copy documents, physical specimens, and unstructured data can be listed and described in the NGDS metadata catalog, these data resources are neither web-accessible nor interoperable. For more about metadata, interoperability, and the difference between structured and unstructured data, please see the Data Models, Exchange Methods, and Metadata page.

Data resources can become part of the NGDS by submittal to an existing node or can be self-submitted and self-hosted through the installation and configuration of an NGDS Node-in-a-box built as an NGDS extension on the CKAN platform. CKAN stands for Comprehensive Knowledge Archive Network. CKAN is modular free-and-open-source repository and data portal software. CKAN has created an NGDS extension in the form of a web-accessible interface by which users can submit and manage metadata records and upload files and datasets.

NGDS Nodes and Repositories

An NGDS node is a public, web-accessible interface that makes geothermal-relevant data assets accessible by providing at least one of the following capabilities:

  • Providing information resources with metadata published through an NGDS catalog node (repository).
  • Hosting a web-accessible folder that:
    • Contains NGDS-conformant metadata files.
    • Is registered for harvesting by an NGDS aggregator (simple metadata publisher).
  • Hosting one or more NGDS-conformant web services (WMS, WFS, WCS, etc.) registered in an NGDS catalog (data server).
  • Hosting a CSW 2.0.2 catalog service that offers metadata conforming to the USGIN ISO profile (metadata publisher).

NGDS Node in a Box

The NGDS Node-in-a-Box (NIAB) is a software package designed to enable quick deployment of an NGDS node. The software installation includes a CKAN instance along with NGDS-specific extensions for conformance with NGDS metadata requirements, support for NGDS information exchanges, and publication of data through OGC services using a linked GeoServer instance. The NGDS NIAB CKAN extension provides users with a web-accessible user interface designed to help NGDS contributors submit and manage metadata records.

Installing the NIAB

Instructions on installing and configuring a NIAB instance are in development and will be available here soon. Contributors interested in installing their own NIAB instance can join the conversation at the NGDS CKAN GitHub.

Finding an Existing NIAB

In addition to installing and configuring a specific Node-in-a-box instance, a registered user may contribute data using a public account on an existing Node-in-a-box if permission is granted from the Node-in-a-box owner. The data contributor is responsible for contacting the owner of the NIAB for permission. A listing of available NIABs, owners, and contact information is in development.

Overview: Becoming an NGDS Node

The simplest way to become an NGDS node is to place file-based information resources in a web-accessible location and then submit metadata describing these resources to the NGDS metadata catalog. Various workflows are available to create and submit metadata:

  • Log in to an existing registered instance of the NGDS Node-in-a-Box (NIAB) and add metadata records through the provided interface.
  • Set up and register an NGDS NIAB instance. This requires:
    1. Installing and configuring the NGDS NIAB software.
    2. Registering the NGDS NIAB instance with the NGDS portal.
    3. Creating a metadata record using the interface provided by your NGDS NIAB instance.
  • Deploy some other metadata management tool that provides a CSW interface to NGDS-conformant ISO metadata for registered resources, and registering the catalog service with the NGDS central portal for harvesting.
  • Create NGDS-conformant ISO metadata records using any other tool and place them in a web-accessible folder that is registered with the NGDS portal for harvesting.

How to Submit Your Metadata Records via a pre-installed NGDS Node-in-a-Box (NIAB)

Data providers can publish their data to NGDS via any NGDS Node-in-a-Box instance for which they have an account with data publication permission.

Connect to the NGDS NIAB using a web browser; click Contribute and populate the appropriate forms to submit a metadata record to the catalog. During this process, either upload or specify the web location of an existing web-accessible data resource. Note that uploading data to an instance of the NIAB sacrifices some degree of control over uploaded data and data providers should make appropriate arrangements with the managers of the node that they are publishing data too before uploading.

Create Account and Select Upload Type. To contribute, sign up for an account and click Contribute; if they are not logged in, users will be prompted to do so upon clicking Contribute.

Clicking Contribute leads to the Contribute page; here, users are presented with two options:

  • Individual Upload: Users create a collection and then add one or more metadata records to this collection; each metadata record describes a data resource. During the metadata record creation process, the user can bind the metadata record to the data resource it describes in one of two ways:
    • By providing a hyperlink to the data resource described by a given metadata record
    • By uploading the data resource described by a given metadata record
  • Bulk Upload: Users upload large numbers of datasets simultaneously in a single zip file; metadata records are generated automatically.

To proceed, click either Individual Upload or Bulk Upload; these options are described below.

Individual Uploads

Performing an individual upload is a three-step process:

  • Create a Collection: Create a collection that will contain one or more metadata record(s)
  • Add Data: Add one or more metadata record(s) to the collection; each metadata record describes a data resource, which will be either specified or uploaded in this step
  • Collection Metadata: Provide metadata that applies to all metadata records in the collection

These steps are described further in the following sections

Step 1: Create a Collection

Clicking Individual Upload on the Contribute page leads to the Create a Collection form; as the name implies, populating this form will create a collection for metadata records, which is the first step of the Individual Upload process. The fields on this form are as follows:

  • Title: Collection title
  • URL: The URL at which this collection will be found. This URL is automatically generated when the Title field is populated, but it can be modified manually
  • Description: A description of the collection and the metadata records stored in it
  • Tags: Keywords that can be used to search for this collection and any metadata records stored in it
  • License: The license under which any dataset(s) described by any metadata record(s) within this collection are published; among other things, the license determines the manner in which the dataset described by the metadata in this collection can be reused. Information on supported licenses can be found at the following locations:
    • Creative Commons
    • GNU Free Documentation License
    • Open Data Commons
    • UK Open Government License

When finished, click Next: Add Data.

Step 2: Add Data

Clicking Next: Add Data on the Create Collection form leads to the Add Data form. Populating this form creates a metadata record describing a specified data resource; this metadata record is added to the collection created in Step 1. Fields found on this form regardless of the type of resource described by this metadata record are as follows:

  • Resource: The URL by which the data resource described by this metadata record can be accessed; if a data resource is instead uploaded to the NGDS Node-in-a-Box Repository, this field will be automatically populated
  • Name: The name of the data resource described by this metadata record as it will appear in the metadata catalog
  • Description: A brief description of the data resource described by this metadata record
  • Structured Resource: Data that has been mapped to an interchange format such as an NGDS data exchange content model; if the data in the dataset described by this metadata record has been mapped to an NGDS data exchange content model, specify the content model to which the data has been mapped. Fields specific to this resource type are as follows:
    • Content Model: The NGDS content model to which the data resource described by this metadata record conforms
    • Content Model Version: The NGDS content model version to which the data resource described by this metadata record conforms
    • Distributor: The individual, agency, company, or organization responsible for distributing the data resource described by this metadata record
    • File Format: The file format of the data resource described by this metadata record
  • Unstructured Resource: Data such as plain text, a scanned image, or a sound file that cannot be processed or analyzed by machines without user input. Fields specific to this resource type are as follows:
  • Distributor: The individual, agency, company, or organization responsible for distributing the dataset described by this metadata record
  • Format: The file format of the data resource described by this metadata record
  • Ordering Procedure: The manner in which the data resource described by this metadata record can be procured
  • Distributor: The individual, agency, company, or organization responsible for distributing the data resource described by this metadata record
  • Protocol: The specific web service protocol used by the web service specified as the data resource described by this metadata record.
  • Layer: The label for a given data service when it is viewed as a layer in a GIS application
  • Offline Resource: A data resource that is only available in print format. Fields specific to this resource type are as follows:
  • Data Service: Data that is available as a web service; specify the type of web service. Fields specific to this resource type are as follows:

More information on data resource types is available on the Data Models, Exchange Methods, and Metadata page.

When the appropriate fields have been populated:

  • Click Save & add another to add another metadata record to the collection; this will repeat the steps described in Step 2
  • Click Next: Additional Info to finalize process by adding collection metadata

Step 3: Collection Metadata

Clicking Next: Additional Info leads to the Collection Metadata form. Here, provide metadata that applies to the collection created in Step 1 and the metadata records within it created in Step 2.

The Collection Metadata form includes the following fields:

  • Visibility: Determines whether or not this collection and the metadata record(s) within it are visible to the public
  • Authors: The authors of the data resource(s) described by metadata record(s) in the collection
  • Maintainer: The individual, agency, company, or organization responsible for the data resource(s) described by metadata record(s) in the collection
  • Spatial Location: A rough approximation of the features in the data resource(s) described by metadata record(s) in the collection
  • Category: A description of the type of data resource(s) described by metadata record(s) in the collection
  • URI: A uniform resource identifier for the collection; see the USGIN URI Tutorial for more information
  • Quality: A rough assessment of the quality of the data resource(s) described by metadata record(s) in the collection
  • Lineage: The provenance of data resource(s) described by metadata record(s) in the collection
  • Status: The workflow status of data resource(s) described by metadata record(s) in the collection
  • Publication Date: The date on which data resource(s) described by metadata record(s) in the collection were published
  • Language: The language in which the data resource(s) described by metadata record(s) in the collection are written
  • Dataset Extent: The physical extent of features in the data resource(s) described by metadata record(s) in the collection; click the Draw a rectangle button ()" width="22" />) to draw a box indicating the dataset extent

When done, click Finish. Doing so will finalize this collection and all metadata records within it; these records will be added to the catalog.

Bulk Uploads

Clicking Bulk Upload leads to the Bulk Upload form. Here, users have the opportunity to upload a .ZIP file containing numerous data resources; the Bulk Upload feature will automatically generate metadata records for each data resource in the ZIP file.

To proceed, click the Choose File button on the second row; in the window that appears, locate the desired .ZIP file and click Open. In the Bulk Upload form, click Submit.

Following the successful submission of metadata to a registered instance of the NGDS NIAB, the metadata records will be harvested by the NGDS central portal aggregator (and possibly by other interested catalog operators) and will appear in NGDS catalog searches. Metadata records in this catalog can be used to locate and access the data resources they describe.
